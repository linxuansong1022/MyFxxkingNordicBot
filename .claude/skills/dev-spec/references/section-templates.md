# DEV_SPEC 各章节：模板 + 苏格拉底提问清单

> 撰写每一章前必须阅读对应章节。模板定义输出格式，提问清单定义对话流程。

---

## Chapter 1: 项目概述 (Project Overview)

### 对话前提问清单

**先问，后写。这些问题帮助你了解用户对项目的认知深度：**

```
1. "用一句话描述你的项目是做什么的？"
   → 测试用户能否简洁概括，暴露理解盲区

2. "你觉得这个项目和市面上已有的 [类似产品/框架] 最大的区别是什么？"
   → 引导用户思考差异化定位

3. "如果你只有 30 秒给面试官介绍这个项目，你会说什么？"
   → 帮用户提炼 elevator pitch

4. "这个项目对你来说意味着什么？是学习项目、生产项目、还是面试项目？"
   → 决定文档的侧重方向
```

### 关键决策点

| 决策点 | 需要讨论的方案 | 面试常见追问 |
|--------|--------------|------------|
| 项目定位 | 学习平台 vs 生产工具 vs Demo | "为什么做这个项目？" |
| 技术路线 | 自研 vs 用框架(LlamaIndex/LangChain) | "为什么不直接用 LangChain？" |

### 输出模板

```markdown
## 1. 项目概述
本项目基于 [核心技术] 设计，目标是 [主要目标]。

### 设计理念 (Design Philosophy)
> **核心定位：[一句话定位]**
> [2-3 句展开说明项目存在的意义和独特角度]

#### 1️⃣ [价值主张一]
[说明 + 具体要点]

#### 2️⃣ [价值主张二]
[说明 + 具体要点]

#### 3️⃣ [价值主张三]
[说明 + 具体要点]
```

## Chapter 2: 核心特点 (Core Features & Design Highlights)

### 对话前提问清单

```
1. "你觉得这个项目在技术上最值得说的亮点是什么？说三个。"
   → 看用户能否识别核心特点

2. "如果把 [特点 X] 去掉，项目会变成什么样？"
   → 帮用户理解每个特点的价值

3. "你知道 [特点 X] 的原理吗？比如混合检索，为什么要同时做
    关键词搜索和向量搜索？只做一种不行吗？"
   → 测试技术理解深度，引出方案对比

4. "[特点 X] 这个设计思路在业界是常见的做法吗？
    你见过其他项目/公司用类似的方案吗？"
   → 拓宽视野，建立知识关联
```

### 关键决策点（按典型 RAG/后端项目列举，根据实际项目调整）

| 决策点 | 方案选项 | 历史背景 | 面试追问 |
|--------|---------|---------|---------|
| 检索策略 | 纯向量 / 纯关键词 / 混合检索 | "2020 年前大多只用 BM25，后来 Embedding 模型发展带来了 Dense Retrieval" | "为什么混合比单一更好？什么时候单一反而更好？" |
| 架构风格 | 单体 / 可插拔 / 微服务 | "早期 RAG demo 大多是硬编码的单体脚本" | "可插拔架构的代价是什么？什么时候不值得？" |
| 重排序 | 不排 / Cross-Encoder / LLM Rerank | "传统搜索引擎只有一阶段排序，后来引入 L2 reranker" | "Cross-Encoder 和 Bi-Encoder 有什么区别？" |
| 数据处理 | 定长切分 / 语义切分 / 递归切分 | "最早的做法是按固定字符数切，容易切断句子" | "你的切分策略如何平衡召回率和精度？" |

### 每个特点的写作模式

```markdown
### [特点名称]
本项目在 [环节] 采用了 [策略]，核心思想如下：

- **[具体做法]**：[详细说明]
    - [子细节 1]
    - [子细节 2]

> **💡 为什么选这个方案？**
> - **问题背景**：[这个方案解决什么问题]
> - **替代方案**：也可以用 [方案 B]，但 [方案 B 的劣势]
> - **业界做法**：[知名公司/项目] 也采用了类似策略
```

## Chapter 3: 技术选型 (Technical Design & Implementation)

### 对话前提问清单

**这是最重要的一章，也是面试最常问的部分。每个子组件都需要对话。**

```
1. "我们项目用了 [组件 X]，你知道它是干什么的吗？
    如果不用它，会怎样？"
   → 从用户已知出发

2. "你听说过 [替代方案 A] 和 [替代方案 B] 吗？
    你觉得它们和我们用的 [组件 X] 有什么区别？"
   → 引出方案对比

3. "假设我们要从 [当前方案] 换到 [替代方案]，
    需要改多少代码？为什么？"
   → 引导理解可插拔架构的价值

4. "这个组件如果挂了/变慢了，系统会怎样？
    我们有没有降级策略？"
   → 引导思考错误处理和鲁棒性
```

### 方案对比标准格式

**每个技术选型点必须在对话中展示，在文档中记录：**

```markdown
#### 3.X.Y [选型点名称]

**问题定义**：[一句话说明要解决什么问题]

**历史演进**：
> 在 [当前方案] 出现之前，业界主要使用 [旧方法]。
> 旧方法的主要痛点是 [痛点1]、[痛点2]。
> [时间点/事件] 之后，[当前方案] 逐渐成为主流。

**方案对比**：

| 维度 | 方案 A | 方案 B | 方案 C |
|------|--------|--------|--------|
| 核心思路 | ... | ... | ... |
| 优势 | ... | ... | ... |
| 劣势 | ... | ... | ... |
| 适用场景 | ... | ... | ... |
| 复杂度 | 低/中/高 | 低/中/高 | 低/中/高 |

**本项目选型**：采用 **方案 X**。
- **选型理由**：[为什么在我们的场景下选 X]
- **代价与接受度**：[选 X 放弃了什么，为什么可以接受]
- **替换路径**：[如果未来需要换成 Y，需要做什么]

> **🎤 面试话术**
> "我们选择 [X] 而不是 [Y]，是因为 [核心理由]。
>  [Y] 的优势在于 [Y 的优势]，但在我们 [具体约束] 的场景下，
>  [X] 在 [关键维度] 上更匹配我们的需求。
>  如果未来 [场景变化]，我们可以通过 [替换路径] 切换到 [Y]。"
```

### 子章节结构模板

```markdown
### 3.X [组件名称] ([English Name])

**目标：** [一句话目标描述]

#### 3.X.1 [子组件名]

**问题定义**：...

**方案对比**：[对比表]

**本项目选型 + 理由**：...

- **设计思路**: [方案描述]
- **关键抽象**: [接口和契约]
- **实现方案**: [具体实现]
- **配置方式**: [settings.yaml 示例]
- **错误处理**: [降级策略]

> **当前实现说明**：[实际已实现 vs 规划中的内容]
```

### 持久化存储统一说明模板

```markdown
> **📌 持久化存储架构统一说明**
>
> | 存储模块 | 数据库文件 | 用途 | 关键字段 |
```

## Chapter 4: 测试方案 (Testing Strategy)

### 对话前提问清单

```
1. "你之前写过单元测试吗？你觉得测试最大的价值是什么？"
   → 了解用户测试经验

2. "你听说过 TDD（测试驱动开发）吗？它和'写完代码再补测试'有什么区别？"
   → 引出 TDD 的 trade-off 讨论

3. "如果你只能写一种测试（单元/集成/E2E），你会选哪种？为什么？"
   → 引出测试金字塔的讨论

4. "你觉得 100% 的测试覆盖率是好事还是坏事？"
   → 测试深度认知
```

### 关键决策点

| 决策点 | 方案选项 | 历史背景 | 面试追问 |
|--------|---------|---------|---------|
| 测试理念 | TDD / 后补测试 / BDD | "TDD 由 Kent Beck 在 2003 年推广" | "TDD 的缺点是什么？什么时候不适合用？" |
| 测试分层 | 多层金字塔 / 只写 E2E / 只写单元 | "早期很多项目只有 E2E 或没有测试" | "测试金字塔的层次是固定的吗？" |
| Mock 策略 | 全 Mock / 真实依赖 / 混合 | "过度 Mock 曾导致很多'测试通过但线上挂了'的事故" | "什么时候应该 Mock？什么时候不应该？" |

### 输出模板

```markdown
## 4. 测试方案

### 4.1 设计理念：[测试理念名称]

> **💡 为什么选 [TDD/其他]？**
> - **替代方案**：[其他测试理念] — [其 trade-off]
> - **我们的选择理由**：[具体理由]

**核心原则**：
- [原则 1]
- [原则 2]

[测试金字塔 ASCII art]

### 4.2 测试分层策略

#### 4.2.1 单元测试
| 模块 | 测试重点 | 典型测试用例 |
|-----|---------|------------|

#### 4.2.2 集成测试
| 测试场景 | 验证要点 | 测试策略 |
|---------|---------|---------|

#### 4.2.3 端到端测试
**场景 N：[场景名]**
- **测试目标**：
- **测试步骤**：
- **验证要点**：

### 4.X 覆盖率目标
- 单元：≥ 80%
- 集成：关键路径 100%
- E2E：核心场景 100%
```

## Chapter 5: 系统架构与模块设计 (System Architecture)

### 对话前提问清单

```
1. "如果让你画这个项目的架构图，你会画几层？每层负责什么？"
   → 测试整体架构认知

2. "数据从用户输入到最终返回结果，经过了哪些模块？"
   → 引导梳理数据流

3. "你觉得哪些模块之间的耦合最紧？如果改了 A，哪些模块可能受影响？"
   → 引导理解模块依赖关系

4. "现在的目录结构，你觉得有没有不合理的地方？"
   → 激发批判性思考
```

### 输出模板

```markdown
## 5. 系统架构与模块设计

### 5.1 整体架构图

> **💡 架构分层的设计理由**
> - 为什么分 [N] 层而不是 [M] 层？
> - 每层之间的依赖方向是 [方向]，这是因为 [原因]

[ASCII 架构图，使用 ┌ ┐ └ ┘ │ ─ ├ ┤ ┬ ┴ ┼ ▼ ▲ 字符]

### 5.2 目录结构

project-name/
├── [目录]                    # [职责说明]
│   ├── [文件]                # [文件职责]
```

### 架构图规范
- 从上（外部调用层）到下（存储层）分层
- 层与层之间标注数据流协议
- 用边框将相关组件分组
- 每一层用一个大框包围

## Chapter 6: 项目排期 (Project Timeline)

### 对话前提问清单

```
1. "你打算花多长时间完成这个项目？有没有 deadline？"
   → 决定排期粒度

2. "你觉得哪个部分最难？最容易卡住？"
   → 帮助合理分配时间

3. "你倾向于先写完所有模块再集成，还是先打通一条最简路径？"
   → 引出"先闭环还是先完善"的 trade-off

4. "每个阶段做完后，你希望看到什么样的'可验证结果'？"
   → 引导定义验收标准
```

### 关键决策点

| 决策点 | 方案选项 | Trade-off |
|--------|---------|----------|
| 开发策略 | 先闭环再完善 / 逐模块开发 / 并行开发 | 闭环能早发现集成问题，逐模块更好测试 |
| 排期粒度 | 1h/任务 / 半天/任务 / 天/任务 | 粒度越细越好追踪，但管理开销越大 |
| 验收方式 | TDD 每步验收 / 里程碑验收 / 最终验收 | TDD 成本高但质量好 |

### 输出模板

```markdown
## 6. 项目排期

> **排期原则**
> - [原则 1]
> - [原则 2]

### 阶段总览

| 阶段 | 目的 |
|------|------|
| A | [目的] |
| B | [目的] |

### 进度跟踪表

> **状态说明**：`[ ]` 未开始 | `[~]` 进行中 | `[x]` 已完成

| 任务编号 | 任务名称 | 状态 | 完成日期 | 备注 |
|---------|---------|------|---------|------|

### [TaskID]：[任务名称]
- **目标**：[一句话]
- **修改文件**：
- **验收标准**：
- **测试方法**：`pytest -q path/to/test.py`

### 交付里程碑
- **M1（完成阶段 X+Y）**：[里程碑描述]
```

## Chapter 7: 可扩展性与未来展望 (Extensibility & Future Vision)

```
1. "如果给你无限时间，你最想给这个项目加什么功能？"

    你会怎么回答？"
   → 面试预演

4. "你了解 [行业趋势，如 Agentic RAG / GraphRAG] 吗？
    觉得你的项目可以往那个方向演进吗？"
   → 拓宽视野
```

```markdown
### 7.1 [方向一] ([English Name])
[当前状态 + 为什么需要演进]
```

## 附录：面试高频追问模式

在任何章节的对话中，都可以穿插这些面试预演：

| 追问类型 | 示例 | 考察意图 |
|---------|------|---------|
| **为什么不用 X** | "为什么不直接用 LangChain？" | 独立思考能力 |
| **规模变化** | "如果数据量增长 100 倍？" | 系统设计能力 |
| **故障场景** | "如果 Embedding API 挂了？" | 工程鲁棒性 |
| **成本约束** | "如果要把 API 调用成本降 80%？" | 成本意识 |
| **团队协作** | "如果 3 个人同时开发这个项目？" | 架构可扩展性 |
| **技术深度** | "RRF 的 k 参数是什么意思？" | 算法理解深度 |
| **横向对比** | "和 [竞品] 比，你的优势是？" | 行业视野 |
