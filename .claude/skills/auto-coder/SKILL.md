# Auto Coder (MiniCode edition)

One trigger drives the full doc-driven feature flow that was taught in the onboarding session:
**Reconnoiter → DEV_SPEC first → Code → Type-check → STUDY_LOG → Verify**.

Optional modifiers:
- Append an inline task (e.g. `auto code /save 命令`) to override the task source
- `--no-commit` to skip the final commit prompt

---

## Project Context (MiniCode)

| Aspect | Value |
|---|---|
| Language | TypeScript (ESM), run via `tsx` — **no build step** |
| Type check | `npx tsc --noEmit` (primary verification, mandatory) |
| Tests | None yet. If a `test/` dir exists later, run whatever's in `package.json` scripts. Do NOT invent a framework |
| Config | `src/config.ts` (`RuntimeConfig`, loaded from 4 layers) — never hardcode values |
| Doc source of truth | `DEV_SPEC.md` at repo root |
| Task source | `STUDY_LOG.md` 最后一条「下一步」section |
| Coding style | Immutable updates (`x = [...x, item]`, not `x.push`). Small files (<800 lines). See `~/.claude/rules/common/coding-style.md` |

**Never** reintroduce any of the following (they're Python-project leftovers):
- `.venv` activation, `pytest`, `config/settings.yaml`, `tests/unit/`
- `[ ]` / `[x]` task markers in DEV_SPEC (MiniCode does NOT use a schedule chapter)

## Pipeline

```
0. Sync Spec  →  1. Reconnoiter  →  2. DEV_SPEC First  →  3. Code
                                                              ↓
                                  6. Verify ← 5. STUDY_LOG ← 4. Type-check
```

Steps 0–5 run autonomously. Step 6 (commit + end-to-end verification) ALWAYS pauses for user confirmation.

---

### 0. Sync Spec

```bash
python3 .claude/skills/auto-coder/scripts/sync_spec.py
```

If `DEV_SPEC.md` hasn't changed since last sync, the script no-ops and prints `specs up-to-date`. Add `--force` to rebuild anyway. The script writes chapter files into `references/` and stores a hash at `.spec_hash`.

---

### 1. Reconnoiter (read-only)

Before touching any code, answer the three scoping questions on paper (or in the response):

1. **Where does data come from? Where does it get consumed?**
2. **Do the producer and consumer know about each other?** — if not, you need a middle module (like `usage-tracker.ts` was for `/cost`)
3. **How many modules will this feature touch?** — enumerate them explicitly

Then do the actual recon:

1. **Find a reference implementation**. Never design from scratch — grep for something similar that already works. Examples:
   - Adding a tool? → read `src/tools/count-lines.ts`
   - Adding a slash command? → read `src/cli-commands.ts`
   - Adding cross-module shared state? → read `src/usage-tracker.ts`
   - Wiring an adapter? → read `src/gemini-adapter.ts`'s `recordUsage` block
2. **Read every file you plan to modify**, end-to-end. TypeScript's compiler will not catch "you forgot to update the second call site". Your brain must.
3. **Identify the insertion line** in each target file (note the line number).

If reconnaissance reveals the feature is bigger than expected, STOP and report scope back to the user — do not proceed.

---

### 2. DEV_SPEC First

**Update `DEV_SPEC.md` BEFORE writing any code.** This is a hard rule — it's in `memory/feedback_doc_driven.md`.

Typical updates:
- New module → add a new `### 4.N` subsection under chapter 4
- Changing an existing module → edit its `### 4.N` subsection (design decisions table, data flow diagram, limitations)
- New slash command → add an entry in chapter 9
- New config key → add an entry in chapter 7

Writing the DEV_SPEC content forces you to:
- Confirm every field/type/edge case (if you can't describe it, you don't understand it)
- Expose "当前局限" you're knowingly accepting (write them down, don't hide them)
- Surface design trade-offs in a table

If writing the spec reveals a problem you hadn't noticed, fix the design NOW (cheapest moment). Do NOT plough ahead into code.

After editing, re-run `python3 .claude/skills/auto-coder/scripts/sync_spec.py --force` so reference files stay in sync.

---

### 3. Code

Order of edits — follow this regardless of feature:

```
底座 (shared infrastructure / types / schemas)
  ↓
写入方 (producers — adapters, event sources)
  ↓
消费方 (consumers — CLI commands, UI, readers)
```

Rules:
- **Match existing patterns**. If similar code uses `async/await` callbacks, don't switch to generators. If other modules use module-level singletons, don't bolt on a class.
- **Immutable updates only** (`records = [...records, x]`, never `push`).
- **No hardcoding** — pull from `RuntimeConfig` or a constant at the top of the file.
- **Use `Read` on files before `Edit`-ing**. Required by the harness and prevents stale-edit mistakes.
- **Tool calls (LLM-facing) go in `src/tools/`. Internal shared state goes in `src/` top level.** If you're about to put non-LLM infrastructure into `src/tools/`, STOP — that's a category error.

---

### 4. Type-check (after every logical unit)

```bash
npx tsc --noEmit
```

Run **after every edited file**, not after the whole batch. If a batch of 5 edits produces 10 errors, you'll waste time locating which edit broke what. `tsc --noEmit` is free — use it aggressively.

If errors appear:
1. Read the first error carefully — errors cascade, so fixing the first often eliminates several
2. Apply minimal fix
3. Re-run `tsc`
4. Repeat

Max 3 auto-fix rounds per file. If still failing, STOP and report to user with the error output.

If a `test/` directory + a `test` script exists in `package.json`, also run `npm test` after `tsc` passes. Otherwise skip tests (project has none).

---

### 5. STUDY_LOG

Append a new dated entry to `STUDY_LOG.md`. The skeleton:

```md
## YYYY-MM-DD

### 学习内容：<one-line task summary>

<2-4 sentences on what changed and why>

### 🌟 最大认知突破

<the non-obvious thing learned — not "I changed files X/Y/Z", but
 "I discovered that X's Y field has semantics unlike Z's">

### 关键认知

<the generalizable rule extracted from this feature>

### 卡点 / 疑问

- ✅ / ⏳ items

### 下一步

- <the next task, so future sessions have a starting point>
```

**Rules for writing STUDY_LOG**:
- Do NOT list "modified files" — `git log` already has that
- DO record "为什么这么改" and踩的坑 — git log doesn't
- Surface any **new** conceptual insight (not a rehash of what was already known)
- Set the 下一步 so the next session can pick up without re-reading everything

---

### 6. Verify & Persist (pauses for user)

Print a summary and STOP:

```
✅ <feature name> — implementation complete
   Files touched:
     - src/foo.ts
     - src/bar.ts
     - DEV_SPEC.md §4.X
     - STUDY_LOG.md (new entry)
   Type check: npx tsc --noEmit → clean
   Pending user verification:
     1. End-to-end: <concrete command the user should run>
     2. Expected: <what a correct result looks like>

Options:
  "verified"  → commit the changes (feat: <msg>)
  "fix: ..."  → apply further fixes before commit
  "skip"      → leave uncommitted, end here
```

**Never commit without user confirmation**, even if tsc is clean. End-to-end verification is the user's job — TypeScript doesn't know whether the feature is "correct", only whether the types line up.

On "verified", create one commit using the project's commit style (from `~/.claude/rules/common/git-workflow.md`):

```
<type>: <short description>

<body explaining why, not what>
```

No Co-Authored-By line (attribution disabled globally per that rules file).

## When NOT to use this skill

- **Pure reading / explaining** tasks — just answer directly, no pipeline
- **Single-file tool additions** (like adding `count_lines`) — the 6-stage flow is overkill; just follow the tool-writing pattern in `src/tools/count-lines.ts`
- **Refactors inside one module** with no cross-module effect — no need for the DEV_SPEC step unless design intent changes
- **Exploratory spikes** where the user explicitly wants to "try something and see" — document-driven flow kills exploration velocity

Use this skill only when: **the task touches ≥2 modules AND changes externally observable behavior**.
